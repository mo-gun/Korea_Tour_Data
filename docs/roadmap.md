### 제안서가 약속한 기능 vs 현재 구현

| 기능 | 제안서 약속 | design/ 목업 현황 | 남은 일 |
|---|---|---|---|
| **M1** 맞춤 동선 추천 | 종목×체류×테마 + 실제 이동시간 | ✅ 엔진 v0 동작 (회복 룰 분기) | 실데이터 연결, 이동시간(모빌리티) 반영 |
| **M2** 마라톤 통합 캘린더 | 전국 300+ 대회, 필터, 출처 표시 | ⚠️ 샘플 7개, 리스트뷰만 | 크롤러 구축, 캘린더뷰 구현 |
| **M3** 인근 축제·행사 | KTO 행사 데이터 지도 표시 | ❌ 정적 2개 하드코딩 | KTO 행사 API 연동 |
| **M4** 러닝·산책 코스 | 두루누비 3~10km 코스 | ❌ 스텁 (샘플만) | 두루누비 API 연동 |

### ⚠️ 최우선 리스크 3가지

1. **KTO OpenAPI 미사용**: 공모전 **필수 요건**인데, 현재 목업은 POI를 전부 카카오맵 검색으로 가져옴. KTO API(01 국문관광정보 / 17 두루누비 / 25 웰니스)를 실제 데이터 경로에 넣어야 함.
2. **마라톤 실데이터 0건**: M2의 "전국 300+ 대회"는 크롤러가 없으면 불가능. `normalize_races.py`는 준비됐지만 입력 CSV를 만드는 크롤러가 없음.
3. **카카오 모빌리티 리드타임**: "실제 이동시간 계산"은 별도 REST 키 발급 + 한도 협의 필요. 지금 문의하지 않으면 8월에 막힘.

## Phase 0 — 분석·의사결정 (7/2 ~ 7/10, 1주)

구현을 시작하기 전에 팀이 확정해야 할 것들.

### 0-1. 기술 스택 확정 ⭐ 가장 큰 결정
- **프론트엔드**: `design/`(Vite+React)을 정식 프론트로 승격 (최신 UX 합의물, 화면 완성도 최고). `web/`(Next.js)은 `types.ts`·`recommend.ts` 로직만 이식하고 아카이브.
- **백엔드**: 두 가지 중 택 1
  - **A안 — Spring Boot + PostgreSQL(+PostGIS) + Redis**: `docs/architecture.md` 설계와 일치, 공공 API 배치·캐싱에 강함. 팀에 Java 역량이 있을 때.
  - **B안 — 경량 서버 (FastAPI/Node)**: 이미 backend/가 Python이므로 FastAPI가 자연스러움. 12주 안에 확실히 완성 가능. 아키텍처 문서와의 차이는 "MVP 우선" 논리로 설명.
  - 판단 기준: **10월에 돌아가는 것이 문서 정합성보다 중요**. 팀 역량으로 결정.

### 0-2. 추천 알고리즘 명세서 작성 (심사 대비 + 구현 기준)
- `design/src/lib/runninggu/engine.js`(v0) + `web/lib/recommend.ts`를 기준으로 문서화:
  - 종목(5K/10K/하프/풀) × 체류(당일/1박/2박) × 테마(6종) 조합별 시간 슬롯 규칙
  - 회복 룰(RECOVERY) 근거 정리 — 러닝 커뮤니티/스포츠의학 자료 인용 붙이기
  - POI 선정 기준(거리순 → 향후 스코어링), 폴백 체인(live/sample/synth)
- 심사에서 "추천 로직이 뭐냐"는 질문에 대한 답이 됨.

### 0-3. API 키·계정 준비 (승인 대기 시간이 있으므로 즉시)
- [ ] 공공데이터포털: KTO **01 국문관광정보 / 17 두루누비 / 25 웰니스** 활용신청
- [ ] 카카오 developers: JS 키(맵), REST 키(로컬) 발급 + 도메인 등록
- [ ] **카카오 모빌리티 길찾기: 별도 키 발급 + 트래픽 한도·비용 문의 시작** (안 되면 폴백: 직선거리×도로계수 추정)
- [ ] 카카오 로그인은 MVP 선택사항 — 저장 기능을 localStorage로 시작하면 뒤로 미룰 수 있음

### 0-4. 역할 분담·마일스톤 합의
- 데이터 파이프라인 / 백엔드 API / 프론트 연동, 3트랙으로 병렬 진행 가능
- `web/docs/DATA_SPEC.md`가 크롤링팀↔프론트팀 계약 문서 — 그대로 사용

## Phase 1 — 데이터 파이프라인 (7/6 ~ 7/24, 3주) ⭐ 크리티컬 패스

M2·M3·M4의 성패가 여기 달려 있음. Phase 2와 병렬 진행.

### 1-1. 마라톤 크롤러 (M2의 원천)
- [ ] 마라톤온라인(marathon.pe.kr)·마라톤GO 크롤러 작성 → `normalize_races.py`가 기대하는 CSV 스키마로 출력
  - 필드: race_id, name, sido, venue, event_date, has_5k/10k/half/full, start_time, reg_status, source, last_checked, detail_url…
- [ ] `normalize_races.py` 실행 → 정규화 + 중복 제거
- [ ] `geocode.py`로 좌표 부여 (KAKAO_REST_KEY 필요) → `races.normalized.json`
- [ ] `races.unresolved.json` 수동 보정 프로세스 확립 (매주 반복할 것이므로)
- **목표: 300+ 대회, 최소 마지노선 100+ (하반기 대회 위주)**

### 1-2. KTO POI 수집기 (공모전 필수 요건 충족의 핵심)
- [ ] KTO API 클라이언트 작성 (공통: serviceKey, MobileOS, MobileApp, _type=json)
  - **01 국문관광정보**: locationBasedList — 대회 좌표 반경 10~15km, contentTypeId 12(관광지)/39(음식점)/15(축제·기간필터)/32(숙박)
  - **17 두루누비**: courseList — 코스명/거리(crsDstnc)/난이도(crsLevel)/GPX 경로
  - **25 웰니스**: locationBasedList — 온천·스파(EX050100) 등 테마코드 활용
- [ ] 대회별 POI 번들 생성 (batch) 또는 서버 실시간 프록시 — Phase 0 스택 결정에 따름
- [ ] `design/`의 PRESAMPLED 구조(경주 1개)를 전 대회로 확장하는 형태가 자연스러움

### 1-3. 갱신·검증 체계
- [ ] 갱신 주기 결정 (권장: 주 1회 배치 + checked 날짜 갱신)
- [ ] DATA_SPEC.md의 검증 체크리스트를 스크립트화 (필수 필드, 좌표 범위, 날짜 형식, 중복)
