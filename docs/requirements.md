# 1. 서비스 정의

## 1.1 한 줄 정의

마라톤 대회를 축으로, 참가 **종목의 완주 피로도에 따라** 대회 전후(D-1/D-day/D+N) 여행 동선을 자동 설계하고 **지도+타임라인**으로 보여주는 러닝 관광 모바일 웹앱.

## 1.2 핵심 차별점 (제안서 약속)

1. **대회→여행 원스톱**: 대회 선택 즉시 인근 맛집·관광지·축제·숙소가 하나의 동선으로 연결된다.
2. **러너 컨디션 기반 추천**: 종목(5K/10K/하프/풀)별 회복 룰이 동선의 강도·구성·순서를 바꾼다.
3. **편집 가능한 초안**: 생성된 동선은 강제가 아니라 초안 — 교체·삭제·추가·순서변경이 자유롭다.
4. **데이터 신뢰 투명화**: 대회 정보의 출처·최근 확인일, POI의 데이터 소스(KTO/카카오/샘플)를 화면에 표기한다.

## 1.3 타겟 사용자

2030 러너(5K~하프 중심). 페르소나 "김러너": 28세 직장인, 10km 55분, 러닝크루와 지방 하프 원정 — 대회 정보는 흩어져 있고, 완주 후 무리하지 않으면서 즐길 동선이 미리 짜여 있길 원함.

## 1.4 제안서 기능 ↔ 앱 매핑

| 제안서 | 정의 | 담당 화면/모듈 |
|---|---|---|
| **M1** 맞춤 동선 추천 | 종목×체류×테마 → 동선 자동 생성 | Plan→Taste→Stay→Result + 엔진 |
| **M2** 마라톤 통합 캘린더 | 전국 대회 검색·필터·상태 | Home |
| **M3** 인근 축제·행사 | 대회 기간 겹치는 행사 표시 | Race(상세), Result(지도) |
| **M4** 러닝·산책 코스 | 두루누비 기반 회복런 코스 | Courses + 동선 내 산책 블록 |

# 2. 사용자 플로우 (전체 시나리오)

```
S1 홈(캘린더)          대회 탐색: 검색·필터 → 대회 카드 선택
   ↓
S2 대회 상세           일정·종목·접수·출처 확인, 인근 축제 미리보기
   ↓ [이 대회로 동선 만들기]
S3 일정 패턴           전날부터 / 대회+다음날 / 전후로 / 당일치기
   ↓
S4 종목·취향           참가 종목(단일) + 여행 취향(복수) → 회복 강도 안내
   ↓
S5 숙소                대회장 주변 숙소 선택 (건너뛰기 가능)
   ↓ [동선 추천받기] → 엔진 실행
S6 동선 결과           지도(핀+경로) ↕ 스크롤 동기화 타임라인, 날짜 탭, 회복 배지
   ↓ [편집]            블록 교체·삭제·추가·순서변경 (후보 시트)
   ↓ [저장]
S7 내 여행             저장된 동선 목록 → 재열람
S8 러닝코스            (대회와 독립) 여행지 근처 코스 탐색            ← 탭바에서 직접 진입
```

- 탭바(4): 캘린더(S1) / 동선(S6) / 러닝코스(S8) / 내 여행(S7). 나머지 화면은 스택 네비게이션(뒤로가기).

## 3.6 FR-ENGINE — 동선 생성 엔진 (M1 핵심) ⭐

> 근거 코드: `design/src/lib/runninggu/engine.js`, `constants.js` — 아래 규칙이 서비스의 심장이며, 백엔드 이관 시에도 동일 규칙을 유지한다.

### 회복 룰 (FR-ENGINE-01, M, ✅)

| 종목 | 권장 도보 상한(walk) | 고강도 제외(noHard) | 피로도 표기 | D-day 방침 | D+ 방침 |
|---|---|---|---|---|---|
| 5K | 8km | ✕ | 거의 정상 | 완주 후 오후부터 자유 관광 | 일반 관광 자유 |
| 10K | 8km | ✕ | 낮은 피로 | 완주 후 가벼운 관광·축제 | 일반 관광 |
| 하프 | 5km | **○** | 중등도 피로 | 완주 후 온천·휴식 권장 | 온천+짧은 산책(고강도 제외) |
| 풀 | 3km | **○** | 고강도 회복 필요 | 완주 후 회복 집중, 도보 최소 | 스파·온천 중심, 도보 최소 |

### POI 후보 구성 (FR-ENGINE-02, M, ✅)

- 카테고리 풀 = `{맛집, 관광지} ∪ 선택 취향`. `noHard` 종목이면 **웰니스 자동 추가**, 아니면 카페 추가.
- 각 카테고리는 **대회장 중심** 8곳, 산책(walk) 후보는 **숙소 중심**(미선택 시 대회장) 자연 카테고리 6곳.
- 동일 장소 중복 배치 금지(이름 기준). 취향 소진 시 폴백 순서: 선택취향 → 관광지 → 자연 → 카페 → 역사.

### 일자별 블록 생성 규칙 (FR-ENGINE-03, M, ✅)

| 구분 | 시간표 (시각 · 블록 · 카테고리) |
|---|---|
| **D-1 (off<0)** | 15:00 숙소 체크인(lodging) → 18:30 카보로딩 저녁(food, "탄수화물 보충·무리 없는 메뉴") → 20:00 가벼운 저녁 산책(walk) · 노트 "내일 완주 · 가볍게 먹고 푹 쉬기" |
| **D-day, noHard** (하프·풀) | 대회 `startTime`(기본 08:00) 🏁 스타트(race) → 11:00 온천·회복(wellness) → *(하프만)* 14:30 가벼운 관광(tour) → 18:00 회복 저녁(food, "소화 잘 되는 회복식") → 20:30 숙소 주변 저녁 산책(walk) |
| **D-day, 일반** (5K·10K) | 🏁 스타트 → 13:00 오후 자유 관광(취향 우선) → 15:30 카페 한 잔(cafe) → 18:30 맛집 저녁(food) → 20:30 저녁 산책(walk) |
| **D+N (off>0)** | 08:00 아침 산책(walk) → 10:00 온천·족욕(wellness, noHard) 또는 오전 관광(tour) → 12:30 로컬 점심(food) → 14:30 오후 관광(취향) → *(마지막 날만)* 17:00 체크아웃·귀가(lodging) |

- 산책 블록 거리 라벨: POI 설명에 km가 있으면 그 값, 없으면 "3.0km 내외 · 쉬움"(회복 상한과 3km 중 작은 값).
- **회복 배지** (FR-ENGINE-04, M, ✅): `noHard` 종목에서만 생성. D+일이 있으면 "D+1 회복 모드", 당일뿐이면 "D-day 회복 모드" + 방침 문구 + 피로도.
- 각 블록은 안정적 ID(`blk_n`)를 가진다 — 편집 시 기존 블록 ID 유지.

# 4. 데이터 요구사항

## 4.1 데이터 모델 (목업 기준 확정 스키마)

**Race(대회)** — 원천 snake_case → `normalizeRace()`로 camelCase 변환:
`id, name, region, venue, date(YYYY-MM-DD), startTime(HH:MM), eventTypes[](풀/하프/10K/5K 표준화·우선순위 정렬), regStatus(접수중|마감), regStart, regEnd, organizer, source(출처), checked(최근 확인일), officialUrl, detailUrl, imageUrl, lat, lng, category`

**POI(장소)**: `name, lat, lng, desc, addr, url` (+ 소스 태그 live|sample|synth|kto)

**Day(일자)**: `date, off(대회일 기준 정수), label(D-1|D-day|D+N), dateLabel, note, blocks[]`

**Block(일정 블록)**: `id(안정적), time(HH:MM), title, catKey(race|lodging|food|tour|cafe|wellness|nature|history|walk|recovery), place(POI|null), desc`

**Trip(저장된 여행)**: `id(race.id+기간), race, raceName, region, event, themes[], start, end, days[], recovery{label,text,intensity}|null`

> 종목 표준화 규칙(`events.js`): 원문(full/42.195/Half/10k/5km…) → `풀|하프|10K|5K` 4종. 그 외 종목(울트라 등)은 제외.

# 7. 제약사항·전제

1. **공모전 요건**: 한국관광공사 OpenAPI 사용 필수 → EI-01~03과 DR-02는 협상 불가. 기능심사(2026-10)에서 실동작 시연.
2. **일정**: 개발 마감 사실상 9월 말. `docs/roadmap.md`의 컷 우선순위(로그인 → 동반자 → 이미지 내보내기) 적용.
3. **아키텍처**: 프론트는 `design/`(Vite+React) 승격 방침. 도메인 로직(`lib/runninggu/`)은 UI 비종속 순수 모듈로 유지 — 백엔드 이관·테스트 용이성의 전제. ⚠️ 루트 CLAUDE.md는 `web/`을 데모 앱으로 기술 — 팀 확정 필요(오픈 이슈 #1).
4. **카카오 심사 리드타임**: 모빌리티 키·한도 협의, (도입 시) 싱크·채널 승인 기간을 일정에 반영.
