## 1. 서비스 개요

| 항목 | 내용 |
|------|------|
| 서비스명 | **런닝구(區)** |
| 슬로건 | "내가 뛸 동네를 가장 잘 아는 친구" |
| 한 줄 정의 | 전국 마라톤 통합 캘린더 + 종목·체류·테마 맞춤 여행 동선 추천 서비스 |
| 타깃 | 2030 러너 (5K/10K/하프 중심). 페르소나 "김러너" |
| 부문 | 관광데이터 활용 공모전 — 웹/앱 개발 |

**문제 정의.** 마라톤 일정은 여러 사이트에 흩어져 있고, 대회 참가 = 사실상 "원정 여행"인데도 종목별 몸 상태(완주 후 피로도)를 고려한 여행 동선을 짜주는 서비스가 없다. 런닝구는 *대회 일정 탐색 → 종목·체류·테마 입력 → 회복까지 고려한 D-1/D-day/D+1 여행 동선* 을 한 번에 제공한다.

**차별점.** 단순 "주변 관광지 나열"이 아니라, **종목별 회복 강도 룰**을 반영해 동선의 강도·순서를 바꾼다 (예: 풀코스는 도보 최소·스파 중심, 5K는 오후 자유 관광).

## 2. 핵심 기능 명세 (제안서 매핑)

### M1 · 종목·상황별 맞춤 동선 추천 *(핵심 기능)*
- **입력:** 대회 1개 선택 + `종목`(5K/10K/하프/풀) × `체류`(당일치기/1박2일/2박3일) × `테마`(관광지/맛집/웰니스/자연/역사유적)
- **처리:** `buildItinerary()` 가 종목별 회복 룰 + 테마 우선순위로 POI를 골라 타임라인 생성 (4번 참고)
- **출력:** D-1 → D-day → D+1 타임라인 + 지도 위 경로(점선) + 마커. 타임라인 항목 클릭 시 지도 포커스 이동
- 담당 컴포넌트: `MarathonDetail`(m1 탭) + `lib/recommend.ts`

### M2 · 전국 마라톤 통합 캘린더
- **입력:** `지역` / `월` / `종목` / `접수가능만` 필터
- **출력:** 조건에 맞는 대회 카드 목록 (대회명·날짜·지역·종목·접수상태 배지·**출처·최근확인일** 표시)
- 신뢰성: 데이터 출처(`src`)와 확인일(`checked`)을 카드에 노출 → 크롤링 데이터의 투명성 확보
- 담당 컴포넌트: `MarathonList`

### M3 · 대회 인근 축제·행사
- 대회 일정·반경 기준 **동시 개최**되는 축제를 지도 마커 + 목록으로 제공
- 담당: `MarathonDetail`(m3 탭). 데이터: `pois` 중 `cat:"festival"`

### M4 · 두루누비 산책 코스
- 거리·난이도별 걷기/회복런 코스. 대회 전 컨디션 조절 또는 비(非)대회 여행에도 활용
- 담당: `MarathonDetail`(m4 탭). 데이터: `pois` 중 `cat:"course"` (`dist`·`level` 필수)

## 4. 추천엔진 명세 (`buildItinerary`)

### 4-1. 종목별 회복 룰 (`RECOVERY`)

| 종목 | 권장보행 | 피로/강도 | noHard | D-day 오후 | D+1 |
|------|------|------|:---:|------|------|
| 5K | 8km 이내 | 거의 정상 | false | 오후부터 자유 관광 | 일반 관광(자유) |
| 10K | 8km 이내 | 낮은 피로 | false | 가벼운 관광·축제 | 일반 관광 |
| 하프 | 5km 이내 | 중등도 피로 | **true** | 온천·휴식 권장 | 온천+짧은 산책(고강도 트레킹 제외) |
| 풀 | 3km 이내 | 고강도 회복 필요 | **true** | 회복 집중, 도보 최소 | 스파·온천 중심, 도보 최소 |

`noHard=true`(하프·풀)면 D+1을 **회복형**으로, false(5K·10K)면 **일반 관광형**으로 분기.

### 4-2. 타임라인 생성 규칙

- **D-1 (당일치기 제외):** 비브픽업/도착(15:00) → 카보로딩 식사(17:30) → 가벼운 산책(19:00, 종목별 보행상한 `풀 3 / 하프 5 / 그외 8`km 이내 코스 선택) → 취침 알림(21:30)
- **D-day:** 출발(대회 startTime) → 완주·샤워(정오) → 이후 종목 분기
  - **풀:** 회복 집중(웰니스, 냉온교차욕) → 휴식·귀가
  - **하프:** 온천·휴식(웰니스) → 가벼운 관광/맛집
  - **5K·10K:** 오후 자유 관광 → 동시개최 축제 → 지역 맛집 저녁
- **D+1 (2박3일만):** `noHard`면 스파/온천 + (하프는 고강도 제외 짧은 산책) + 정적 관광 / 아니면 오전관광·점심·오후관광 → 체크아웃

### 4-3. POI 선택 로직 (`pickPOI`)
- 카테고리(cat) 내에서 **입력 테마 일치 POI 우선**, 없으면 같은 cat 첫 항목 폴백
- `used[]` 배열로 **이미 쓴 POI 중복 방지** (한 동선에 같은 장소 두 번 안 나옴)

> 엔진은 DOM·지도에 의존하지 않는 **순수 함수**라 데이터 소스가 더미든 실데이터든 그대로 재사용된다.

## 5. 기술 스택 & 아키텍처

| 레이어 | 기술 |
|--------|------|
| 프레임워크 | Next.js 14.2.5 (App Router) |
| 언어 | TypeScript 5.5 |
| 스타일 | Tailwind CSS 3.4 (brand #0f7b6c / accent #ff7a45) |
| 지도 | 카카오맵 JavaScript SDK (CustomOverlay 핀, Polyline 경로) |
| 데이터(현재) | `lib/data.ts` 더미 `MARATHONS` |
| 데이터(목표) | 크롤링 + KTO OpenAPI → **정적 `marathons.json`** (DATA_SPEC 참고) |

**데이터 흐름(목표).**
```
크롤링(마라톤온라인·마라톤GO) + KTO API(국문관광·두루누비·웰니스)
   → 정규화·병합·지오코딩·POI매칭 → public/data/marathons.json
   → 프론트가 읽어 더미 대체 → 추천엔진/UI는 무수정 재사용
```

## 6. 외부 데이터·API

| 용도 | 소스 | 키/비고 |
|------|------|------|
| 마라톤 일정 | marathon.pe.kr, marathongo.co.kr | 크롤링 (공식 API 없음) |
| 관광·맛집·축제 POI | 한국관광공사 국문관광정보 OpenAPI | `KTO_SERVICE_KEY` |
| 산책 코스(M4) | 두루누비 정보 서비스 | KTO |
| 웰니스(회복형) | 웰니스 관광정보 서비스 | KTO |
| 지도 표시 | 카카오맵 SDK | `NEXT_PUBLIC_KAKAO_MAP_KEY` (클라이언트) |
| 좌표 변환(지오코딩) | 카카오 로컬 API | 장소명 → lat/lng |
| 이동시간(선택) | 카카오 모빌리티 길찾기 | 동선 소요시간 표시 |

## 9. 역할 분담 (3명 기준)

| 사람 | 역할 | 산출물 |
|------|------|------|
| 1 | **마라톤 수집** — 크롤링 2소스 정규화·중복병합·지오코딩 | 대회 레코드(POI 제외) |
| 2 | **POI 수집** — KTO 3종 API, 대회별 반경 매칭 | 각 대회의 `pois` |
| 3 | **프론트 + 통합/검증** — 1·2 데이터 합쳐 `marathons.json` 생성, 프론트 연동·화면 | 동작하는 웹 |

- 지오코딩(장소명→좌표)은 1·2 공통 필요 → 공용 함수 하나로 만들어 공유
- 통합·검증(C)은 별도 인력 없이 3번이 겸하거나 셋이 마무리에 함께 점검
