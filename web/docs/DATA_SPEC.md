# 런닝구 데이터 명세 (Data Contract)

수집팀(크롤링·KTO API)이 만들어내는 최종 산출물 `marathons.json`의 형식을 못박은 문서.
**크롤링이든 API든, 결과는 전부 이 모양으로 정규화해서 합친다.** 프론트는 이 파일만 읽는다.

> 단일 진실 소스(Single Source of Truth)는 `web/lib/types.ts` 의 `Marathon` / `Poi` 타입이다.
> 이 문서와 타입이 어긋나면 **타입이 우선**. 스키마를 바꾸려면 타입부터 고치고 이 문서를 갱신한다.

---

## 0. 전체 흐름

```
[크롤링] marathon.pe.kr ─┐
                         ├─→ 정규화 → 중복병합 → 지오코딩 → POI 매칭 → marathons.json → 프론트
[크롤링] marathongo ─────┘                                  ↑
[KTO API] 국문관광·두루누비·웰니스 ──────────────────────────┘ (POI 공급)
```

산출물은 **하나의 JSON 파일** `web/public/data/marathons.json` (배열 `Marathon[]`).
프론트는 빌드/런타임에 이 파일을 읽어 더미(`lib/data.ts`의 `MARATHONS`)를 대체한다.

---

## 1. 최상위 형식

`marathons.json` = `Marathon` 객체의 **배열**.

```json
[
  { "id": "...", "name": "...", "...": "..." },
  { "id": "...", "name": "...", "...": "..." }
]
```

---

## 2. Marathon 필드 명세

| 필드 | 타입 | 필수 | 규칙 / 예시 |
|------|------|:---:|------|
| `id` | string | ✅ | 전역 유일. 영문 소문자+숫자 권장. 예: `"chuncheon-2026"`. 중복병합의 키가 아님(아래 6번 참고) |
| `name` | string | ✅ | 대회 공식명. 예: `"춘천마라톤"` |
| `region` | string | ✅ | 표시용 지역명. `"시/도 + 시군구"` 권장. 예: `"강원 춘천시"` |
| `regionCode` | string | ✅ | 필터용 광역 지역 코드. **고정 목록은 3번 참고.** 예: `"강원"` |
| `date` | string | ✅ | **`YYYY-MM-DD` 고정.** 예: `"2026-10-25"` |
| `month` | number | ✅ | `date`의 월(1~12). `date`와 반드시 일치. 예: `10` |
| `lat` | number | ✅ | 대회장 위도(소수점). 지오코딩 결과. 예: `37.8813` |
| `lng` | number | ✅ | 대회장 경도. 예: `127.7298` |
| `status` | enum | ✅ | `"open"` \| `"soon"` \| `"closed"` (4번 참고) |
| `events` | string[] | ✅ | `"5K"`\|`"10K"`\|`"하프"`\|`"풀"` 중 1개 이상. 순서 무관, 중복 금지 (5번 참고) |
| `bib` | string | ✅ | 비브(배번) 픽업 안내. 모르면 `"대회 홈페이지 참고"` |
| `startTime` | string | ✅ | `"HH:MM"` 24h. 모르면 종목 대표 출발시각. 예: `"08:00"` |
| `src` | string | ✅ | 데이터 출처(표시·신뢰성용). 예: `"마라톤온라인"` 또는 원본 상세 URL |
| `checked` | string | ✅ | 최근 확인일 `YYYY-MM-DD`. 크롤링/검수한 날짜 |
| `pois` | Poi[] | ✅ | 이 대회에 묶인 주변 POI 배열. 비어 있어도 됨(`[]`)이나 가급적 채운다 (7번 참고) |

> `lat/lng`가 없으면 지도에 못 찍힌다. 지오코딩 실패한 대회는 별도 `unresolved.json`으로 빼두고 `marathons.json`엔 넣지 않는다.

## 3. regionCode 고정 목록

필터 칩과 1:1로 매칭되므로 **아래 값만** 쓴다 (수도권·광역시 중심):

```
서울 · 인천 · 경기 · 부산 · 대구 · 대전 · 광주 · 울산 · 세종
강원 · 충북 · 충남 · 전북 · 전남 · 경북 · 경남 · 제주
```

- `region`(표시용)은 자유롭게 상세히("강원 춘천시"), `regionCode`(필터용)는 위 광역 단위로.

---

## 4. status 판정 규칙

크롤링한 접수 상태 원문을 아래로 변환:

| status | 의미 | 판정 기준(예) |
|--------|------|------|
| `open` | 접수중 | 접수 시작일 ≤ 오늘 ≤ 마감일 |
| `soon` | 접수예정 | 오늘 < 접수 시작일 |
| `closed` | 마감 | 마감일 < 오늘, 또는 "접수마감"·"선착순 마감" 표기 |

- 접수일 정보가 없으면 대회일 기준 보수적으로: 대회일이 지났으면 `closed`, 아니면 `open`.

---

## 5. events 매핑 규칙

원문 종목 표기를 **4종으로만** 매핑한다. 그 외 거리는 가까운 종목으로 흡수.

| 원문 예시 | 매핑 |
|------|------|
| `5km`, `5K`, `5킬로`, `건강달리기 5` | `"5K"` |
| `10km`, `10K`, `단축마라톤` | `"10K"` |
| `Half`, `하프`, `21km`, `21.0975K` | `"하프"` |
| `Full`, `풀`, `42.195`, `마라톤(풀코스)` | `"풀"` |
| `3km`, `걷기`, `워킹` 등 초단거리 | 가까운 `"5K"`로 흡수하거나 제외(팀 합의) |

- 같은 대회에 여러 종목이면 배열로: `["5K","10K","하프","풀"]`.

---

## 6. 중복 병합 (두 소스 합치기)

같은 대회가 marathon.pe.kr과 marathongo 양쪽에 나온다.

- **병합 키 = `정규화된 대회명 + date`.**
  - 대회명 정규화: 공백·연도·"제N회"·괄호 부가설명 제거 후 비교. 예: `"제24회 춘천마라톤 2026"` → `"춘천마라톤"`.
- 키가 같으면 한 레코드로 합치고, 더 정보가 풍부한 쪽을 채택. `src`엔 둘 다 표기 가능(`"마라톤온라인, 마라톤GO"`).
- `id`는 병합 후 하나만 부여.

---

## 7. Poi 필드 명세 + 매칭 규칙

POI는 KTO API 3종에서 가져와 **각 대회 위치 반경 안의 것만** 그 대회 `pois`에 넣는다.

| 필드 | 타입 | 필수 | 규칙 |
|------|------|:---:|------|
| `cat` | enum | ✅ | `"food"`\|`"tour"`\|`"wellness"`\|`"course"`\|`"festival"`\|`"race"` |
| `name` | string | ✅ | POI 이름 |
| `lat` | number | ✅ | 위도 |
| `lng` | number | ✅ | 경도 |
| `desc` | string | ✅ | 한 줄 설명 |
| `themes` | string[] | ⬜ | `"관광지"`\|`"맛집"`\|`"웰니스"`\|`"자연"`\|`"역사유적"` 중. 추천엔진이 테마 매칭에 사용 |
| `dist` | number | ⬜ | **course 전용.** 코스 거리(km) |
| `level` | enum | ⬜ | **course 전용.** `"쉬움"`\|`"보통"`\|`"어려움"` |

**cat ↔ KTO 소스 매핑**

| cat | 출처 | 비고 |
|-----|------|------|
| `tour` | 국문관광정보 (관광지) | `themes`에 관광지/자연/역사유적 부여 |
| `food` | 국문관광정보 (음식점) | `themes: ["맛집"]` |
| `festival` | 국문관광정보 (축제·행사) | **대회 주말과 기간이 겹치는 것만** (M3) |
| `course` | 두루누비 (걷기길) | `dist`·`level` 필수 (M4) |
| `wellness` | 웰니스 관광정보 | `themes: ["웰니스"]` |
| `race` | — | 대회장 자체. 보통 마커로만 쓰고 pois엔 안 넣어도 됨 |

**반경 규칙(권장):** 대회장 기준 반경 약 10~15km 이내. `food`·`tour`는 대회장 가까운 순 상위 N개(예: 각 6~10개), `festival`은 기간 겹치는 전부, `course`는 거리순.

## 9. 검증 체크리스트 (JSON 제출 전 자가점검)

- [ ] 최상위가 배열인가
- [ ] 모든 필수 필드가 채워졌나 (특히 `lat`/`lng`/`date`/`regionCode`)
- [ ] `date`가 `YYYY-MM-DD`이고 `month`와 일치하나
- [ ] `regionCode`가 3번 고정목록 안의 값인가
- [ ] `events`가 4종 enum뿐인가, 빈 배열 아닌가
- [ ] `status`가 3종 enum뿐인가
- [ ] `pois[].cat`이 6종 enum뿐인가, `course`엔 `dist`/`level`이 있나
- [ ] `id` 중복 없나
- [ ] 지오코딩 실패 대회를 본 파일에 섞지 않았나

> 가능하면 `web/lib/types.ts`를 그대로 import 하는 검증 스크립트(`tools/validate.ts`)를 만들어 CI에서 돌리면 안전.

## 10. 프론트 연동 (수집팀 → 프론트 핸드오프)

1. 수집팀: 완성된 배열을 `web/public/data/marathons.json`으로 저장·커밋.
2. 프론트: `lib/data.ts`에서 `MARATHONS`를 이 JSON으로 교체 (타입은 그대로 `Marathon[]`).
   - 더미 데이터 시절의 `RECOVERY`/`CAT_LABEL`/`CAT_COLOR`는 유지.
3. 추천엔진(`buildItinerary`)·UI는 **수정 불필요** — 데이터만 갈아끼우면 됨.

---

## 11. 역할 분담 제안 (정하기 나름)

- **A: 마라톤 수집** — 크롤링 2소스 → 정규화·중복병합(5·6번) → `status`/`events` 변환 → 지오코딩(`lat/lng`)
- **B: POI 수집** — KTO 3종 API → 대회별 반경 매칭(7번) → `pois` 채우기
- **C: 통합·검증** — A+B 합쳐 `marathons.json` 생성, 9번 체크리스트/검증 스크립트
- **D: 프론트** — JSON 연동(10번), 화면 다듬기, 카카오 모빌리티(선택)
